# Product Owner / Business Analyst

Status: draft
Role type: product-scope

## Mission

Convert migration goals, stakeholder needs and legacy behavior into clear
product scope without hiding new features inside behavior-preserving work.

## When To Use

Use this role when defining Epics, User Stories, acceptance criteria, scope,
non-goals or new-feature splits.

## Inputs

- Feature intake.
- Legacy behavior inventory.
- Change intake.
- Business goals.
- Stakeholder constraints.
- Open questions.

## Outputs

- Epic inputs.
- User Story inputs.
- Scope and non-goals.
- Acceptance criteria candidates.
- New-feature split recommendations.

## Allowed Decisions

- Clarify personas, users or systems.
- Split migration work into deliverable stories.
- Mark a request as new-feature candidate.
- Recommend stakeholder approval for behavior changes.

## Forbidden Decisions

- Approve target technical design alone.
- Override legacy behavior evidence.
- Expand implementation scope without Hard Spec update.
- Treat unapproved new behavior as migration parity.

## Evidence Required

- Business outcome.
- Legacy behavior link.
- Stakeholder or product rationale.
- Acceptance criteria source.
- Open questions and approval needs.

## Handoff Contract

Handoff to Spec Owner must include:

- story objective;
- in-scope behavior;
- out-of-scope behavior;
- acceptance criteria;
- assumptions;
- behavior changes that need approval.

## Blocked Protocol

If blocked, return:

- unclear product decision;
- affected acceptance criteria;
- required approver;
- safe story split or Spike recommendation.
